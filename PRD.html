01. 项目背景

问题痛点

平台分散

AI对话工具（豆包、Kimi、千问、DeepSeek）分散使用，收藏内容难以统一管理

查找困难

收藏的对话缺乏有效整理，查找历史内容费时费力

信息孤岛

跨平台知识无法关联，形成信息孤岛，难以形成知识体系

效率低下

手动整理耗时费力，无法应对大量对话的整理需求

目标用户

高频使用AI工具的职场人士 需要整理学习笔记的学生/研究者 跨平台使用AI的产品经理/开发者

核心价值主张

"让散乱的AI对话变成结构化的知识资产"

04. AI整理引擎设计

核心策略：免费为主，付费可选

默认配置（全免费）：利用各平台免费额度，90%场景零成本运行

任务 使用模型 原因 成本
简单分类 本地轻量模型 / 规则 快速、免费 免费
摘要生成 通义千问免费版 / Kimi 中文好、免费额度足 免费
复杂推理 DeepSeek V3 推理强、价格低 低成本
高质量输出 GPT-4o-mini / Claude（用户自选） 效果最佳 用户付费

处理流程

1

第一层：规则匹配（本地，免费）

  • • 用户自定义标签规则
  • • 关键词匹配
  • • 正则匹配
2

第二层：轻量AI（免费API）

  • • 通义千问：分类、标签
  • • Kimi：摘要生成
3

第三层：深度AI（低成本/付费）

  • • DeepSeek：复杂推理
  • • GPT-4o-mini（用户自选）：高质量输出

05. 数据流设计

同步策略

首次同步：全量同步

  • • 获取所有历史收藏
  • • 建立本地索引
  • • 完整数据备份

后续同步：增量同步

  • • 基于时间戳获取新增
  • • 支持手动/自动触发
  • • 节省带宽和时间

数据存储

混合存储架构

本地

IndexedDB存储原始对话（隐私保护）

云端

可选同步（跨设备）

缓存

内存缓存热点数据

08. 技术约束

已知限制

平台API不提供收藏读取接口（技术预研发现）

无法直接回写修改原平台数据

应对策略

提供多种数据导入方式

间接回流（生成建议+手动复制）
